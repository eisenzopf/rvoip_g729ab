**Key decisions (resolved in implementation plan):**

1. **DspContext threading model**: All ops take `&mut DspContext` as first parameter. Functions that don't set the overflow flag simply don't touch it, but the uniform signature means callers never need to know which ops can overflow.
2. **`#[inline(always)]`**: All basic ops and DPF ops. Called millions of times per second.
3. **Word16/Word32 newtypes**: `#[repr(transparent)]` wrappers around i16/i32 with Copy, Clone, Debug, Eq, Ord. No operator overloading -- explicit `add(a, b)` calls signal DSP intent.
4. **Omitted functions**: `L_macNs`, `L_msuNs`, `L_add_c`, `L_sub_c`, `mac_r`, `msu_r` are defined in `basic_op.c` but never called in g729ab_v14 -- omitted to avoid unused complexity. `L_macNs`, `L_msuNs`, `L_add_c`, `L_sub_c` are the only operations that use the `Carry` flag, so `DspContext` does not include a `carry` field. `mac_r` and `msu_r` are round-after-MAC variants with no callers in the reference code.
5. **`round()` naming convention**: The C basic operation `round()` (Word32→Word16 rounding: `extract_h(L_add(var1, 0x00008000))`) maps to `dsp::arith32::round()` in Rust. This collides with `f64::round()` if glob imports are used (`use crate::dsp::arith32::*`). Always use qualified `arith32::round()` calls, or alternatively name it `dsp_round()` if glob imports of the arith32 module are desired for ergonomics. The per-phase specs should use the qualified name in all pseudocode.

**Functions to specify (~38):**

- **16-bit (7)**: `add`, `sub`, `mult`, `mult_r`, `negate`, `abs_s`, `saturate`
- **32-bit (13)**: `L_add`, `L_sub`, `L_mult`, `L_mac`, `L_msu`, `L_negate`, `L_abs`, `L_sat` (NOTE: `L_sat` has no callers in the reference code — included for completeness as a utility), `L_deposit_h`, `L_deposit_l`, `extract_h`, `extract_l`, `round`

- **Division/math (4)**: `div_s`, `inv_sqrt`, `Log2`, `Pow2`
- **DPF (5)**: `L_Extract`, `L_Comp`, `Mpy_32`, `Mpy_32_16`, `Div_32`
- **Random (1)**: `Random()` LCG pseudo-random generator

**TDD requirements:**

- 29 boundary tests from implementation plan test table (all in `src/dsp/arith.rs` `#[cfg(test)]`), including 4 `L_deposit_h`/`L_deposit_l` boundary tests (`MAX_16`, `MIN_16` for each)
- Property-based tests via proptest: commutativity (`add(a,b) == add(b,a)`), identity (`add(a,0) == a`), range preservation
- C cross-validation: 10,000 random input pairs per function compared against C `basic_op.c` outputs
- Random() seed verification: FER seed (21845) and CNG seed (11111) sequences match C reference

**C reference fixtures:** `random_fer_seq.bin`, `random_cng_seq.bin` (generated by `tests/scripts/generate_c_dumps.sh`)

- `GPCLIP2 = 481` (Q9): confirmed used by gain pre-selection during taming — `best_gain[0]` is clipped to GPCLIP2 in `QUA_GAIN.C:138-140` when `tameflag == 1` (Phase 6 gain quantization)
- Validation method for each table (CRC32 checksum, spot checks, automated parser)
- Module file mapping from implementation plan: `tables/window.rs`, `tables/lsp.rs`, `tables/gain.rs`, `tables/pitch.rs`, `tables/misc.rs`, `tables/bitstream.rs`, `tables/postfilter.rs`, `tables/vad.rs`, `tables/sid.rs`
- `tables/sid.rs` must include `noise_fg_sum[MODE][M]` (pre-computed row sums of `noise_fg`, used by `sid_lsfq_decode` in `dec_sid.c:179-180` and `lsfq_noise` in `qsidlsf.c:101,115`) and `noise_fg_sum_inv[MODE][M]` (pre-computed inverse row sums, used by `lsfq_noise` in `qsidlsf.c:241,250`). These are defined in `tab_dtx.c` and are required for both SID encoding (Phase 8) and SID decoding (Phase 9)
- **Annex B DTX sizing constants** (`#[cfg(feature = "annex_b")]`): `SIZ_ACF = 22` (= `NB_CURACF * (M+1)`, buffer size for current autocorrelation frames), `SIZ_SUMACF = 33` (= `NB_SUMACF * (M+1)`, buffer size for summed autocorrelation frames). Defined in `dtx.h:85-86`, used by DTX state buffers in Phase 8. These belong in `tables/sid.rs` alongside other Annex B constants. Cross-reference: Phase 8 content outline documents their usage context
- **`noise_fg` derivation comment requirement:** Since `tables/sid.rs` replaces the C runtime function `Init_lsfq_noise()` (`dec_sid.c`) with compile-time `const` arrays, the file must include a derivation comment for `NOISE_FG[1]` values documenting the exact formula: `// Derived from fg[]: NOISE_FG[1][i][j] = (fg[0][i][j] * 19660 + fg[1][i][j] * 13107) >> 15` with the note that this uses truncation (via `extract_h`), NOT rounding. `NOISE_FG[0]` is an identity copy of `fg[0]`. This comment enables future maintainers to audit or regenerate the table from `fg` without consulting the C source
